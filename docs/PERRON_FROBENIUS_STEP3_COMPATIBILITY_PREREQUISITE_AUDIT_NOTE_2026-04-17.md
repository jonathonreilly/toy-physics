## Question

Before a hard-review-safe step-3 Perron-Frobenius compatibility theorem can be
attempted honestly on this branch, what exact prerequisites must already be in
hand?

## Answer

A genuine step-3 common-state compatibility theorem is well-typed only after
four distinct prerequisites are closed:

1. **Common parent object.**  
   One canonically axiom-derived positive parent operator or positive-map
   object must be fixed on the science surface where sector comparison is meant
   to happen.

2. **Sector descendant laws.**  
   For each live sector to be compared, there must be an exact descendant,
   projection, or intertwiner law from that parent object into the sector
   transfer object.

3. **Fixed sector PF states.**  
   Each sector must contribute one determined Perron-Frobenius state on an
   already fixed descendant object, not merely a family with framework-point
   underdetermination.

4. **Matched step-3 hypotheses.**  
   Only after those exact objects and maps exist may positive-map PF theory be
   used as a theorem template, because only then is common-state comparison
   mathematically well-posed.

On the current exact stack:

- item 1 is only partially closed: the Wilson gauge surface has a real parent
  object, but there is not yet one global parent theorem across the live
  sectors;
- item 2 is still open on the PMNS side: there is no Wilson-to-PMNS
  descendant / intertwiner theorem;
- item 3 is still open on the plaquette side: the framework-point `beta = 6`
  operator-side / Perron data are not yet determined;
- item 4 therefore remains unavailable too.

So the present branch state does **not** yet support an honest step-3 theorem
search on concrete common-state data. The compatibility front should currently
audit hypotheses, not pretend closure.

## Theorem 1: exact prerequisite audit for the step-3 compatibility theorem

On the current exact stack, a hard-review-safe step-3 compatibility theorem
requires simultaneous closure of the following exact prerequisites:

1. one common positive parent object at the level where sector comparison is
   intended;
2. one exact descendant / projection / intertwiner law for each sector that is
   supposed to inherit that parent structure;
3. one determined sector PF state on each such descendant object;
4. one matched positive-map PF hypothesis package on those exact objects.

But the present theorem bank still has:

1. only a partial parent theorem on the Wilson gauge surface rather than one
   global parent theorem across the live sectors;
2. no Wilson-to-PMNS descendant / intertwiner theorem;
3. no determined plaquette framework-point `beta = 6` operator-side / Perron
   data;
4. and therefore no honest match yet to the load-bearing step-3 positive-map
   hypotheses.

Therefore the current branch does **not** merely lack the final step-3 proof.
It still lacks the exact prerequisite package that would make the step-3
comparison problem theorem-ready.

## Corollary 1: sector-local PF exactness is not yet a common-state theorem

The branch already has real sector-local PF / dominant-mode theorems.

But that is still weaker than:

- one common parent PF state,
- exact sector descendant maps from that parent,
- and one fixed sector PF state on each live descendant object.

So sector-local exactness should not be narrated as if the common-state problem
were already typed.

## Corollary 2: the compatibility scout front should audit hypotheses, not import rhetoric

The honest step-3 scout task at current stack level is:

1. name the exact parent object that would carry the comparison,
2. name the exact sector maps that would transport that object into plaquette
   and PMNS,
3. verify that the sector PF states are fixed rather than underdetermined,
4. only then check external PF hypotheses.

That is the review-safe scope for step-3 work today.

## Corollary 3: the first positive step-3 note must be typed object-by-object

Any future positive step-3 theorem note should explicitly name:

- the parent operator or positive map,
- the sector projection / intertwiner laws,
- the fixed plaquette framework-point operator data,
- the exact PMNS descendant object,
- and the matched positive-map PF hypotheses.

Without that typing, the branch is only restating the aspiration of step 3 and
not proving it.
