## Authority disclaimer

This is a backward-looking synthesis. Every claim about a named row's
audit state, claim_type, criticality, score, or repair-class prefix is
sourced from the live `docs/audit/data/audit_queue.json` and
`docs/audit/data/audit_ledger.json` at write time and re-verified by

## Scope

A row is `ready=True` in [`docs/audit/data/audit_queue.json`](audit/data/audit_queue.json)
when its one-hop dependencies are all retained-grade, so the next Codex
audit pass is allowed to pick it up. At write time the live queue
holds **36 ready rows** across all criticality tiers. The synthesis
classifies each into one of six unlock-pattern categories, lifted from
patterns visible across recent narrow-source-note PRs:

- **A — clean-just-waiting:** will audit clean on the next pass; no
  source-note edit required.
- **B — misclassification candidates:** `claim_type` set to a value
  that no longer matches the source note's structural shape; canonical
  example is the alpha_lm reclassification ([PR #969](https://github.com/jonathonreilly/cl3-lattice-framework/pull/969)).
- **C — data-bug candidates:** ledger field carries a stale or
  malformed value (e.g. file-path-as-claim_id); canonical example is
  Diamond ([PR #771](https://github.com/jonathonreilly/cl3-lattice-framework/pull/771)).
- **D — terminal-blocked candidates:** prior `audited_conditional`
  verdict carries a structural repair class
  (`runner_artifact_issue` etc.) that will not repair as a small
  bounded source-note edit.
- **E — open-gate candidates:** `claim_type=open_gate`; would audit
  `audited_clean` `open_gate` (recognized open work).
- **F — chain-blocked candidates:** deps retained-grade now BUT the
  source note's load-bearing step depends on a not-yet-cited
  authority that needs to be wired (Wilson-target-citation pattern,
  [PR #955](https://github.com/jonathonreilly/cl3-lattice-framework/pull/955)).

## Recommendation for future campaigns

The single highest-leverage cheap unlock currently on the queue is
the B candidate alpha_lm reclassification, already in flight as
[PR #969](https://github.com/jonathonreilly/cl3-lattice-framework/pull/969)
(score 17.711, critical, ~413 transitive descendants gated). No
further B candidates are visible on the current queue.

The A category (4 rows) is straightforward: schedule the next Codex
audit pass to pick them up; they should land clean. No source-note
edits required.

The C category is empty; no Diamond-pattern data-bug fix is
currently available.

The D category (29 rows) is the heaviest pile. **Future
audit-backlog-campaign / retained-promotion-campaign cycles should
not spawn against any D row without first lining up the named
artifact-recovery / runner-classification / runner-registration
work** — the audit lane will deterministically return the same
conditional verdict and cycles are wasted. The repair-class
sub-table above shows where to invest if any of these scores
becomes high-leverage downstream.

The F category (2 rows) is similarly chain-blocked by missing
upstream authorities. Per the per-leaf deprioritization synthesis
notes (PRs #960, #984), the F rows are currently terminal-blocked
in the bounded-source-note pipeline; future repair would require
deriving and seeding the named authority as its own retained row
before re-citing.

The 31 rows that are NOT independently attackable as fresh-audit
candidates (D + F) collectively represent ~86% of the live ready
queue. The forward-motion budget on this surface is concentrated in
the 5 A+B rows (~14% of the queue) — and one of those (alpha_lm)
already has its reclassification in flight.
